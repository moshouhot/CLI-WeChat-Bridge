# 版本 0.9.0

**发布日期：** 2026 年 3 月 27 日

## 概览

版本 0.9.0 重点解决本地 companion 工作流的启动成本和 bridge 生命周期问题，让微信侧与本地终端的协同更加稳定。这个版本新增了 Codex 和 Claude 的一键启动命令，引入了带生命周期约束的临时 bridge，明确强化了 shell 模式的非交互远程执行定位，并修复了附件发送与状态同步中的多个边界问题。

完整的 `0.8.0..0.9.0` tag 差异共涉及 43 个文件，新增 5,515 行、删除 154 行。其中包含本次新增的 `docs/releases/` 历史发布文档归档。如果不计算 release 文档，代码与测试层面的变化为 28 个文件，新增 2,691 行、删除 154 行。

### 关键统计
- **完整 tag 差异共 43 个文件**

## 1. Codex / Claude 一键 Companion 启动

在此前的推荐流程中，Codex 和 Claude 都需要手动开两个终端，一个运行 bridge，另一个运行可见 companion。0.9.0 为两条路径都补上了单命令入口，可以直接为当前目录拉起或复用正确的 bridge，等待本地 endpoint 就绪后再打开可见 companion。

### 新命令

- `wechat-codex-start`
- `wechat-claude-start`

### 行为变化

- 如果当前目录已经有对应 bridge 在运行，则直接复用
- 如果现有 bridge 正在服务其他目录或其他适配器，则自动停止并替换
- 只有在当前工作区 endpoint 真正可达后，才会打开可见 companion
- 后台 bridge 以 `companion_bound` 生命周期运行，所以关闭可见 companion 后，这个临时 bridge 也会自动退出

### 支持参数

```bash
wechat-codex-start --cwd D:\work\my-project
wechat-codex-start --profile work
wechat-claude-start --cwd D:\work\my-project
wechat-claude-start --profile work
```

- `--cwd <path>`
- `--profile <name-or-path>`
- `--timeout-ms <ms>`

### 对应仓库内开发命令

- `bun run codex:start`
- `bun run claude:start`

原有的双终端手动启动方式依然保留。这两个新命令只是把相同架构封装成了更快的启动入口。

## 2. Bridge 生命周期与退出可靠性

0.9.0 明显收紧了 bridge 的所有权模型与退出行为，尤其针对本地 companion 驱动的临时 bridge 工作流。

### 生命周期感知

- `wechat-bridge` 新增 `--lifecycle <persistent|companion_bound>` 参数
- bridge lock 现在会记录生命周期与父进程 PID
- 当父进程已经退出时，可以自动回收遗留的临时 bridge
- companion 断开连接后，临时 bridge 会主动请求关闭

### 更干净的退出路径

- bridge 现在会监听父进程退出、终端关闭、stdin close/end、`SIGINT`、`SIGTERM`、`SIGHUP` 和 `SIGBREAK`
- 关闭前会尽量刷新待发送的文本和附件队列
- 修复 Codex 可见 panel 正常关闭后 bridge 仍然残留的问题
- 微信侧收到的 fatal error 会被收敛成更短、更可读的用户提示，而不是把最近的 app-server 日志整段抛进聊天

### 本地 loopback 代理绕过

- 会确保 `127.0.0.1`、`localhost`、`::1` 被加入 `NO_PROXY` / `no_proxy`
- 避免 Codex / Claude 的本地 bridge 流量被系统或企业代理错误接管

## 3. Shell 模式与附件发送工作流改进

### Shell 模式明确定位为非交互远程执行器

- shell bridge 现在明确面向非交互命令和脚本，不再把 REPL 或全屏交互程序视为正常工作流
- 对裸 `python`、`node`、`powershell`、`bash`、`cmd`、`ssh`、`vim`、`tmux` 等交互式调用会直接拒绝，并返回可执行的替代建议
- 通过每条命令独立的 completion sentinel 跟踪结束状态，过滤 prompt 回显噪音，并把 exit code 干净地回传到微信

### 附件发送更可靠，也更少误触发

- 只有当微信入站消息看起来确实是在要求“把本地文件或媒体发回微信”时，才会注入附件协议提示
- 恢复普通本地文件的自动发送能力，不再局限于图片、视频、音频等媒体类型
- 对多重扩展名文件、家目录相对路径、以及包裹在 URL 中的本地文件路径有了更稳定的解析
- 普通叙述中提到的源码文件和脚本路径不会再被误判成要上传到微信的附件

## 4. Claude 与 Codex 状态同步修复

### Claude 的 compact / resume 处理更稳定

- `/compact` 的完成状态现在既能从 PTY 输出识别，也能从结构化 session 事件识别
- compact 失败时会正确回传失败信息并把 bridge 恢复到 idle，而不是让微信侧 turn 卡死
- 如果持久化的 Claude transcript 已被 compact 删除，bridge 会丢弃过期的 resume 元数据并直接启动新会话，而不是启动时报错
- 当 Claude 会话 compact 完成后，微信侧会收到明确提示，说明 bridge 已经可以继续接收新消息

### Codex 退出行为更合理

- 在 panel 模式下，Codex 原生可见窗口以 `exit code 0` 退出会被视为正常结束
- 正常 panel 关闭过程中，后续 websocket / app-server 的 fatal error 会被抑制
- 降低了可见 Codex panel 正常关闭时的误报噪音

## 重要说明

- 现有用户不需要做配置迁移。
- 手动双终端工作流 `wechat-bridge-codex` + `wechat-codex`、`wechat-bridge-claude` + `wechat-claude` 仍然可用。
- `shell` 模式现在应被视为非交互执行器。请优先使用 `cmd /c ...`、`powershell -Command ...`、`bash -c ...`、`python script.py`、`node script.js` 这类一次性命令或脚本入口，而不是启动交互式 shell 或 REPL。
- 如果你希望 bridge 自动跟随当前目录，并在关闭可见 companion 时一起退出，优先使用 `wechat-codex-start` 和 `wechat-claude-start`。
