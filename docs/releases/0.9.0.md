# Version 0.9.0

**Release Date:** March 27, 2026

## Overview

Version 0.9.0 focuses on making the local companion workflow easier to start and safer to keep aligned with WeChat. This release adds one-click starter commands for Codex and Claude, introduces lifecycle-aware transient bridges, hardens shell mode as a non-interactive remote executor, and fixes several attachment and state-sync edge cases across Codex and Claude.

## 1. One-Click Local Companion Startup for Codex and Claude

Previously, the recommended Codex and Claude workflows required manually starting the bridge in one terminal and the visible companion in another. Version 0.9.0 adds dedicated one-click starters that can launch or reuse the correct bridge for the current directory, wait for the local endpoint, and then open the visible companion automatically.

### New commands

- `wechat-codex-start`
- `wechat-claude-start`

### What they do

- Reuse an existing bridge when it is already serving the current directory
- Stop and replace a bridge that is serving a different directory or adapter
- Wait for the current workspace endpoint to become reachable before opening the visible companion
- Start the bridge in a transient `companion_bound` lifecycle so closing the visible companion also stops that temporary bridge

### Supported options

```bash
wechat-codex-start --cwd D:\work\my-project
wechat-codex-start --profile work
wechat-claude-start --cwd D:\work\my-project
wechat-claude-start --profile work
```

- `--cwd <path>`
- `--profile <name-or-path>`
- `--timeout-ms <ms>`

### Related repo-local scripts

- `bun run codex:start`
- `bun run claude:start`

The original two-terminal flows still work. These new commands are a faster wrapper around the same local companion architecture.

## 2. Bridge Lifecycle and Shutdown Reliability

Version 0.9.0 makes bridge ownership and shutdown behavior much stricter, especially for transient local-companion workflows.

### Lifecycle-aware bridges

- Adds `--lifecycle <persistent|companion_bound>` to `wechat-bridge`
- Records lifecycle and parent PID in bridge lock files
- Automatically reclaims orphaned transient bridges whose parent process has already exited
- Stops transient bridges when the visible companion disconnects

### Cleaner shutdowns

- Watches parent process exit, terminal close, stdin close/end, `SIGINT`, `SIGTERM`, `SIGHUP`, and `SIGBREAK`
- Flushes pending text and attachment send queues before shutting down
- Prevents Codex bridges from lingering after the native panel closes
- Converts internal fatal transport messages into shorter user-facing WeChat errors instead of dumping recent app-server logs into chat

### Local loopback proxy bypass

- Ensures `127.0.0.1`, `localhost`, and `::1` are always added to `NO_PROXY` and `no_proxy`
- Prevents local Codex and Claude bridge traffic from being routed through corporate or system HTTP proxies

## 3. Shell Mode and Attachment Workflow Improvements

### Shell mode is now explicitly non-interactive

- Shell bridge now targets non-interactive commands and scripts, not REPL or full-screen sessions
- Rejects interactive invocations with actionable guidance for cases like bare `python`, `node`, `powershell`, `bash`, `cmd`, `ssh`, `vim`, `tmux`, and similar tools
- Tracks completion with per-command sentinels, strips prompt and echo noise, and reports exit codes cleanly back to WeChat

### Attachment sending is more reliable and less noisy

- Injects the WeChat attachment reply protocol only when an inbound message really looks like a request to send a local file or media back to WeChat
- Restores automatic sending for ordinary local files such as PDFs and text documents, not just media files
- Resolves multi-dot filenames, home-relative desktop/document paths, and wrapped local file URLs more reliably
- Avoids accidentally uploading source code and script paths that are mentioned in ordinary prose

## 4. Claude and Codex State Sync Fixes

### Claude compact and resume handling

- Detects `/compact` completion from both PTY output and structured session-start events
- Detects compact failures from terminal output and returns the bridge to idle instead of leaving WeChat turns hanging
- Drops stale Claude resume metadata when a persisted transcript was deleted by compaction, then starts a fresh session instead of crashing on startup
- Sends a clear WeChat notice when a Claude conversation has been compacted and the bridge is ready for new messages again

### Codex exit handling

- Treats a clean native Codex panel exit (`exit code 0`) as expected in panel mode
- Suppresses follow-on websocket and app-server fatal errors during clean panel shutdown
- Reduces false fatal noise when the visible Codex panel closes normally

## Important Notes

- No config migration is required for existing users.
- The manual two-terminal flows (`wechat-bridge-codex` + `wechat-codex`, `wechat-bridge-claude` + `wechat-claude`) continue to work.
- `shell` mode should now be treated as a non-interactive executor. Use commands like `cmd /c ...`, `powershell -Command ...`, `bash -c ...`, `python script.py`, or `node script.js` instead of starting interactive shells or REPLs.
- If you want the bridge to follow the current directory automatically and stop with the visible companion, prefer `wechat-codex-start` and `wechat-claude-start`.
